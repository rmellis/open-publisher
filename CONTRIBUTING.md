# Contributing to OpenPublisher

Thank you for your interest in OpenPublisher! We appreciate the time and effort of everyone who wants to help improve the project.

Currently, OpenPublisher is maintained by a very small core team of two people. Because our bandwidth for reviewing and maintaining new code is strictly limited, we are highly selective about the Pull Requests (PRs) we accept. We want to ensure that every addition aligns perfectly with the project's long-term vision and doesn't introduce future maintenance overhead.

To save everyone time, please read through the following guidelines before making any changes.

## How You Can Help

### 1. Bug Reports and Feature Requests
The best way to contribute right now is by thoroughly documenting bugs or proposing well-thought-out features. 
*   **Bugs:** Use the issue tracker to report bugs. Please include clear steps to reproduce, your operating system, and the version of OpenPublisher you are using if it's the App.
*   **Features:** If you have an idea for a feature, open an issue to discuss it. Please explain *why* this feature is crucial and how it benefits the broader user base.

### 2. Code Contributions
We hold a very high standard for merging community code. We generally only accept contributions that represent significant, high-impact improvements and are fully aligned with our immediate roadmap. 

If you believe you have an addition that meets this bar, **you must follow this process:**

1.  **Open an Issue First:** Don't start writing code or submit a Pull Request without discussing it with the maintainers first. Open an issue detailing your proposed changes, the implementation strategy, and why it is essential. 
2.  **Wait for Approval:** Wait for explicit approval and a "go-ahead" from a core maintainers before working on the PR. PRs submitted without prior discussion will likely be closed.
3.  **Code Quality:** If approved, your code must match the existing style of the project, be extensively commented where complex, and avoid introducing unnecessary dependencies.
4.  **Expect Thorough Reviews:** Even approved concepts will undergo strict code review. Be prepared for requested changes or the possibility that the PR may ultimately not be merged if it doesn't meet the project's stability requirements.

### 3. Spread the Word
Building a community of users is just as important as writing code. Sharing OpenPublisher with others, writing tutorials, or providing feedback on your workflow are incredibly valuable ways to support the project.

Thank you for understanding our process and helping us keep OpenPublisher focused and stable!
